# GKE Private Clusters: Master Authorized Networks & Custom Subnetworks

![GKE Private Cluster Architecture](./images/gke-private-cluster-architecture.png)

**A comprehensive implementation guide for secure Kubernetes cluster deployment with network isolation, master authorized networks, and custom subnetwork configurations.**

---

## Overview

This repository documents the implementation of Google Kubernetes Engine (GKE) private clusters with advanced security configurations. The lab demonstrates how to:

- Create private clusters with no public IP addresses on nodes
- Configure Master Authorized Networks to restrict control plane access
- Implement custom subnetworks with dedicated primary and secondary CIDR ranges
- Establish secure connectivity through bastion hosts
- Verify network isolation and authentication mechanisms

**Security Benefits:**

- Node pools operate without external IP addresses, reducing attack surface
- Kubernetes control plane accessible only from authorized IP ranges
- Google APIs accessible via private Google Cloud APIs (no public internet routing)
- Pod and service traffic isolated within VPC network boundaries

---

## Key Concepts: Auto-Generated vs Custom Subnetworks

| Feature | Auto-Generated Subnet | Custom Subnet |
|---------|----------------------|---------------|
| **Creation** | Automatic during cluster creation | Manual creation required |
| **Primary Range** | Automatically assigned | User-defined (e.g., 10.0.0.0/22) |
| **Secondary Ranges** | Auto-generated by GKE | User-defined (pods and services) |
| **Control** | Limited customization | Full control over CIDR allocation |
| **Use Case** | Quick prototyping and testing | Production environments with network policies |
| **Name Pattern** | gke-[cluster-name]-subnet-[hash] | User-provided name |
| **Flexibility** | Single cluster per subnet | Multiple clusters can share subnets |

## Implementation Guide

### Prerequisites

- Google Cloud account with billing enabled
- gcloud CLI installed and configured

## Task 1: Environment Initialization & Configuration

Initialize the environment with project, region, and zone configurations.

```bash
# Verify project configuration
gcloud config get-value project

# Set compute zone
gcloud config set compute/zone us-east1-d

# Export variables for scripting
export REGION=us-east1
export ZONE=us-east1-d
```

**Purpose**: Establishes the working context for all subsequent GKE operations.

---

## Task 2: Private Cluster Creation (Default Subnet Auto-Generation)

Create a private cluster with auto-generated subnetwork and IP aliases enabled.

```bash
gcloud beta container clusters create private-cluster \
  --enable-private-nodes \
  --master-ipv4-cidr 172.16.0.16/28 \
  --enable-ip-alias \
  --create-subnetwork "" \
  --machine-type e2-medium
```

### Flag Reference

| Flag | Purpose |
|------|---------|
| `--enable-private-nodes` | Nodes receive only private IP addresses |
| `--master-ipv4-cidr 172.16.0.16/28` | CIDR block for master control plane (must be /28) |
| `--enable-ip-alias` | Enable IP aliasing for pod/service ranges |
| `--create-subnetwork ""` | Auto-create subnetwork with default network |
| `--machine-type e2-medium` | VM instance type for node pool |

**Wait for cluster creation to complete** (approximately 3-5 minutes).

## Task 4: Securing the Control Plane

### Step 1: Create Bastion Host VM

```bash
gcloud compute instances create source-instance \
  --zone=$ZONE \
  --machine-type=e2-medium \
  --scopes 'https://www.googleapis.com/auth/cloud-platform'
```

### Step 2: Get External IP Address

```bash
EXTERNAL_IP=$(gcloud compute instances describe source-instance --zone=$ZONE --format="get(networkInterfaces.accessConfigs.natIP)")
echo "Bastion Host IP: $EXTERNAL_IP"
```

### Step 3: Authorize External Access

```bash
gcloud container clusters update private-cluster \
  --enable-master-authorized-networks \
  --master-authorized-networks "$EXTERNAL_IP/32"
```

### Step 4: Install kubectl and Configure Access

```bash
# SSH into bastion host
gcloud compute ssh source-instance --zone=$ZONE

# Inside SSH session:
sudo apt-get update
sudo apt-get install kubectl -y
sudo apt-get install google-cloud-sdk-gke-gcloud-auth-plugin -y

# Configure cluster access
gcloud container clusters get-credentials private-cluster --zone=$ZONE

# Verify nodes (no external IPs)
kubectl get nodes --output yaml | grep -A4 addresses
kubectl get nodes --output wide
```

**kubectl get nodes --output yaml (excerpt):**

## Task 5: Advanced Isolation via Custom Subnetworks

### Step 1: Create Custom Subnet with Secondary Ranges

```bash
gcloud compute networks subnets create my-subnet \
  --network default \
  --range 10.0.4.0/22 \
  --enable-private-ip-google-access \
  --region=$REGION \
  --secondary-range my-svc-range=10.0.32.0/20,my-pod-range=10.4.0.0/14
```

### Step 2: Create Second Private Cluster Using Custom Subnet

```bash
gcloud beta container clusters create private-cluster2 \
  --enable-private-nodes \
  --enable-ip-alias \
  --master-ipv4-cidr 172.16.0.32/28 \
  --subnetwork my-subnet \
  --services-secondary-range-name my-svc-range \
  --cluster-secondary-range-name my-pod-range \
  --zone=$ZONE \
  --machine-type e2-medium
```

### Step 3: Authorize Access for Second Cluster

```bash
# Get new external IP (bastion VM IP remains the same)
EXTERNAL_IP=$(gcloud compute instances describe source-instance --zone=$ZONE --format="get(networkInterfaces.accessConfigs.natIP)")

gcloud container clusters update private-cluster2 \
  --enable-master-authorized-networks \
  --zone=$ZONE \
  --master-authorized-networks "$EXTERNAL_IP/32"
```

### Step 4: Verify Second Cluster

```bash
gcloud compute ssh source-instance --zone=$ZONE

# Inside SSH session:
gcloud container clusters get-credentials private-cluster2 --zone=$ZONE

# Verify nodes
kubectl get nodes --output yaml | grep -A4 addresses
kubectl get nodes --output wide
```

### Custom Subnet CIDR Summary

| Component | Range | Description |
|-----------|-------|-------------|
| Primary | 10.0.4.0/22 | Node IPs |
| Pods (Secondary) | 10.4.0.0/14 | Pod IPs |
| Services (Secondary) | 10.0.32.0/20 | Service IPs |
| Master | 172.16.0.32/28 | Control Plane |

## Task 6: Clean Up

Remove all created resources to avoid unnecessary charges.

```bash
# Delete second cluster
gcloud container clusters delete private-cluster2 --zone=$ZONE

# Delete first cluster
gcloud container clusters delete private-cluster --zone=$ZONE

# Delete source VM
gcloud compute instances delete source-instance --zone=$ZONE

# Delete custom subnet
gcloud compute networks subnets delete my-subnet --region=$REGION

# Optional: Return to default network settings
gcloud compute networks subnets delete --quiet $(gcloud compute networks subnets list --network default --format="value(NAME)") --region=$REGION
```

## Troubleshooting

### Common Issues

| Issue | Solution |
|-------|----------|
| Cluster creation timeout | Check quota limits for `inUseAddresses` and `instanceTemplates` |
| Master authorized network rejection | Ensure CIDR includes source IP with /32 prefix |
| kubectl connection refused | Verify VM has `cloud-platform` scope and network tag allows SSH |
| Subnet range conflicts | Ensure primary range doesn't overlap with existing networks |

### Diagnostic Commands

```bash
# Check cluster status
gcloud container clusters list

# View cluster details
gcloud container clusters describe private-cluster

# Check authorized networks
gcloud container clusters describe private-cluster --format="value(masterAuthorizedNetworksConfig.cidrBlocks)"

# Verify node configuration
kubectl get nodes -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{range .status.addresses[*]}{.type}{"="}{.address}{" "}{end}{"\n"}{end}'
```
